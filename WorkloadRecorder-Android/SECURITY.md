# 安全策略

## 报告安全漏洞

如果您发现了安全漏洞，请通过以下方式报告：

1. **通过GitHub Issues** (请避免在公开Issue中披露漏洞细节)
2. **通过私人沟通渠道** (如果已建立)

### 报告时请提供

- 漏洞的简要描述
- 影响版本
- 复现步骤
- 预期行为
- 修复建议(如果有)

## 安全最佳实践

### 开发者

#### 密钥安全

✅ **必须做的**
- 使用GitHub Secrets管理所有密钥
- 不在代码中硬编码任何密钥
- 轮换任何可能泄露的密钥
- 使用密码管理器生成强密钥

❌ **禁止做的**
- 提交密钥库、API密钥等敏感文件
- 在公共仓库中包含生产环境密钥
- 在日志中打印敏感信息

#### 依赖安全

✅ **必须做的**
- 定期更新依赖项
- 使用Dependabot自动检查更新
- 审查所有新添加的依赖
- 只使用官方和可信的源

❌ **禁止做的**
- 使用已知有漏洞的依赖版本
- 添加未经审查的第三方库
- 忽略安全警告

### 运维

#### CI/CD安全

- 最小权限原则配置GitHub Actions
- 定期轮换访问令牌
- 审查所有工作流变更
- 保护敏感构建工件

#### 发布安全

- 始终签名发布版本
- 验证发布的完整性
- 记录每个发布的变更
- 提供回滚计划

## 已知安全考量

### WebView安全

- WebView配置了安全设置
- JavaScript Bridge方法已保护
- 输入已进行适当清理
- 不允许不安全的内容加载

### 数据存储

- 用户数据存储在本地
- 敏感信息不会发送到外部服务器
- 数据备份加密保护
- 应用沙箱保护

## 安全更新

我们承诺及时处理安全问题：

- 关键安全漏洞：48小时内响应
- 高风险问题：1周内修复
- 中等问题：1个月内修复
- 低风险问题：下一个常规版本中修复
